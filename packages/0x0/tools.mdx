Tools are the actions that agents can invoke during a session. Each tool has a defined schema,
permission requirements, and output format.

## Tool registry

0x0 maintains an internal tool registry. Provider SDKs (Claude Code, Codex) use their own tool
names which are mapped to registry IDs for permission checks.

### Built-in tools

| Registry ID | SDK action names | Description |
| --- | --- | --- |
| `bash` | `Bash` | Execute shell commands with configurable timeout |
| `read` | `Read` | Read file contents with optional offset and line limit |
| `write` | `Write` | Write content to a file (creates or overwrites) |
| `edit` | `Edit`, `MultiEdit`, `NotebookEdit` | Make targeted string replacements in files |
| `search` | `Glob`, `Grep` | Search files by name (glob mode) or content (grep mode) |
| `search_remote` | `WebFetch`, `WebSearch` | Web search, URL fetch, or code search |
| `apply_patch` | `ApplyPatch` | Apply a unified diff patch to one or more files |
| `task` | `Task` | Delegate work to a sub-agent (handoff or background) |
| `todo_write` | `TodoWrite` | Create and update structured TODO lists |
| `question` | `AskUserQuestion` | Ask the user interactive questions with options (only available for `app`, `cli`, and `desktop` clients) |
| `lsp` | `Lsp` | Language Server Protocol operations (experimental, requires `experimental.lsp_tool: true`) |
| `docs` | `Docs` | Context7 documentation search (experimental, requires `experimental.context7_api_key`) |

### Agent actions

Agent `actions` config uses the **SDK action names** (PascalCase), not registry IDs. For example:

```yaml
agent:
  reviewer:
    actions:
      Read: allow
      Glob: allow
      Grep: allow
      WebFetch: allow
      WebSearch: allow
      Task: allow
      AskUserQuestion: allow
```

The legacy `tools_allowed` field uses registry-style IDs (`bash`, `read`, `search`, etc.) and is
automatically converted to the `actions` format internally.

## Permission mapping

Multiple tools can map to the same permission. Configuring a permission affects all mapped tools.

| Permission | Registry tools |
| --- | --- |
| `read` | `read` |
| `edit` | `edit`, `write`, `apply_patch` |
| `search` | `search` |
| `search_remote` | `search_remote` |
| `bash` | `bash` |
| `task` | `task` |
| `todowrite` | `todo_write` |
| `question` | `question` |
| `lsp` | `lsp` |
| `skill` | `skill` |
| `external_directory` | Access to directories outside the project root (checked by `read`, `search`, `edit`, and other file-based tools) |

See [Permissions](/packages/0x0/permissions) for how to configure access rules.

## Legacy tool name migrations

Older `tools_allowed` values are automatically converted to `actions`:

| Legacy `tools_allowed` value | SDK actions |
| --- | --- |
| `bash` | `Bash` |
| `read` | `Read` |
| `search` | `Glob`, `Grep` |
| `search_remote` | `WebFetch`, `WebSearch` |
| `apply_patch` | `Edit`, `Write`, `MultiEdit`, `NotebookEdit` |
| `task` | `Task` |
| `todowrite` | `TodoWrite` |
| `question` | `AskUserQuestion` |

## Custom tools

Custom tools can be added in two ways:

### Plugin tools

Register tools via the `tool` hook in a [plugin](/packages/0x0/plugins):

```typescript
tool: () => ({
  'my-tool': {
    description: 'Does something useful',
    args: { query: z.string() },
    async execute(args, context) {
      return `Result: ${args.query}`
    },
  },
})
```

### Local tool files

Place `.ts` or `.js` files in your project or global tool directory:

- **Project**: `.zeroxzero/tool/*.{ts,js}` or `.zeroxzero/tools/*.{ts,js}`
- **Global**: `~/.zeroxzero/tool/*.{ts,js}` or `~/.zeroxzero/tools/*.{ts,js}`

## Controlling tool access

Use `actions` in agent config to control which tools an agent can use:

```yaml
agent:
  reviewer:
    actions:
      Read: allow
      Glob: allow
      Grep: allow
      AskUserQuestion: allow
```

Only tools with an `allow` or `ask` action are available. Omitting `actions` on native agents
allows all tools by default. The legacy `tools_allowed` field is still supported but `actions`
is preferred.
